**Security controls already in place** (uncommon at this stage):

- Supabase **JWT auth** (JWKS / RS256+ES256) applied at the router level; public routes are the explicit exception.
- **No LLM-generated Cypher** — LightRAG context-only retrieval; the RAG Postgres role is read/write (LLM cache). UI projection access is gated by JWT on `/graph`/chat (backend uses `SUPABASE_DB_URL`).
- **Rate limiting** (`slowapi`) on `/chat`.
- **Generic 500 body** + `request_id`; full trace logged server-side only (no `str(exc)` leakage).
- **Security headers** middleware (`X-Content-Type-Options`, `X-Frame-Options`, HSTS, `Cache-Control: no-store`).
- **CORS allowlist** with a prod fail-fast (`_validate_env` refuses to boot on placeholder / `*` origins in prod).
- **Non-root** backend container + healthcheck; reproducible builds via `uv.lock` + `--frozen`.
- **Postgres checkpointer + store** wired for agent memory/resumability.
- **LangSmith tracing** bridged into the backend process with `run_name` / `tags` / `metadata` for filterable traces.

**Bottom line:** There are **no critical blockers** remaining. The open items are hardening, operational, and documentation gaps — the kind you close out on the way to prod, not architectural rework.

## 3. Fixes applied in this review

F1–F3 were verified with `ruff` + `pyright` + full `pytest` (71 passing). M1–M2 add CI and documentation. All of the above are intended to land together in git when the owner requests a commit.

### F1 — Untracked committed LangGraph dev-state artifacts
- **Files:** `.gitignore`, `apps/agents/.langgraph_api/*`
- **Problem:** Seven pickle files (`*.pckl` — checkpoints, store, vectors, ops, retry counter) generated by `langgraph dev` were **tracked in git**. These are local runtime state (potentially containing conversation data), churn on every dev run, and should never be versioned.
- **Fix:** `git rm -r --cached apps/agents/.langgraph_api` and added to `.gitignore`:
  ```gitignore
  # LangGraph dev server state (regenerated by `langgraph dev`; never commit)
  .langgraph_api/
  *.pckl
  ```

### F2 — Hardened the read-only Cypher guard against false positives
- **File:** `apps/agents/src/agents/safety.py` (+ tests in `apps/agents/tests/test_safety.py`)
- **Problem:** `_WRITE_CLAUSE` was matched against the **raw** query. A legitimate *read* query that merely mentions a write keyword inside a string literal was wrongly rejected — e.g. a movie titled **`Set It Off`** (`\bSET\b`), **`The Drop`** (`\bDROP\b`), or a comment. On rejection, `run_graph_query` fails closed and returns no graph facts, silently degrading answer quality for real films.
- **Fix:** String literals, backtick-quoted identifiers, and comments are blanked out **before** the write-clause scan. Real writes are still rejected (including `CALL { CREATE }`).
- **Tests added:** keyword-in-string-literal allowed, keyword-in-comment allowed, real `SET` next to a benign literal still rejected.

### F3 — Non-root user in the agents Dockerfile
- **File:** `apps/agents/Dockerfile`
- **Problem:** The agents image ran as **root** (the backend image already drops privileges). Also, the container CMD is `langgraph dev` — a development server (see **H1**).
- **Fix:** Added an `appuser` (with `/app` ownership so uv can write its cache) and `USER appuser`, plus a code comment documenting that `langgraph dev` is a dev/Studio server and production should deploy via LangGraph Platform / `langgraph up`.

## 4. Detailed findings

### 🟠 High

#### H1 — Agent container runs a development server as its production command
- **Location:** `apps/agents/Dockerfile:14` (`CMD ["uv", "run", "langgraph", "dev", ...]`)
- **Detail:** `langgraph dev` is the in-memory Studio/dev server; it is not intended to serve production traffic. Two valid resolutions:
  1. **The agent service is dev-only** (Studio), and production uses only `apps/backend`, which imports and compiles the graph directly with the Postgres checkpointer/store (this is how `apps/backend/src/api/main.py` lifespan already works). In that case, do not deploy the `agent` service to prod, and document that.
  2. **You do want a standalone agent server** — then deploy via **LangGraph Platform** or `langgraph up` (production runtime) rather than `langgraph dev`.
- **Recommendation:** Decide the deployment topology and align `docker-compose.yml` + the Dockerfile accordingly. A note was added to the Dockerfile in F3.

#### H2 — `langgraph.json` CORS is fully permissive
- **Location:** `apps/agents/langgraph.json:9-20`
- **Detail:** `allow_methods: ["*"]`, `allow_headers: ["*"]`, `allow_credentials: true`. This is acceptable for a local Studio server but dangerous if the LangGraph server is ever exposed publicly. It is directly coupled to **H1**: if the agent server is dev-only, this is fine; if it's exposed, tighten it.
- **Recommendation:** Restrict methods/headers/origins if the server is reachable outside localhost, or keep the agent server private.

### 🟡 Medium

#### M1 — No CI pipeline — ✅ Fixed
- **Location:** `.github/workflows/ci.yml`
- **Detail:** `pre-commit`, `ruff`, `pyright`, and `pytest` were configured locally, but nothing enforced them in the remote.
- **Fix:** GitHub Actions on push/PR to `main`: `uv sync --frozen` → `ruff check` → `ruff format --check` → `pyright` → `pytest`.

#### M2 — No root README or deployment runbook — ✅ Fixed
- **Location:** `README.md`, `docs/setup/deployment.md`
- **Detail:** A repo about to ship needed a root README and a deploy runbook (env vars, ingestion order, memory/DB setup, secrets, CORS).
- **Fix:** Root `README.md` (quick start + architecture) and `docs/setup/deployment.md` (prod topology, env table, Neo4j/Postgres setup, health, CI, rotation, rollback).

#### M3 — Neo4j driver authenticates as the admin user
- **Location:** `apps/agents/src/agents/clients.py:84-99`
- **Detail:** Read-only is enforced at the application layer (`ensure_read_only` + `execute_read`), and the docstring documents this trade-off. On Neo4j Community/Aura Free a dedicated read-only role isn't available, but on Enterprise/Aura Pro it is — and the workspace's own `security.mdc` rule calls for "a dedicated read-only Neo4j role/user."
- **Recommendation:** In production on a tier that supports it, connect the agent with a read-only role as defense-in-depth (so an app-layer bypass still can't write).

#### M4 — Docker base images are not pinned
- **Location:** `apps/agents/Dockerfile:2,4`, `apps/backend/Dockerfile:2,4`, `apps/frontend/Dockerfile:1`
- **Detail:** `python:3.11-slim`, `node:20-alpine`, and especially `uv:latest` are floating tags → non-reproducible builds and surprise upgrades.
- **Recommendation:** Pin to digests (`python:3.11-slim@sha256:...`) and pin uv to a fixed version tag rather than `latest`.

#### M5 — `/ready` does not actually probe Postgres
- **Location:** `apps/backend/src/api/routes/health.py:17-26`
- **Detail:** Readiness verifies Neo4j via `verify_connectivity()` but only performs an attribute access (`request.app.state.checkpointer`) for the DB layer — a dead pool would still report "ok."
- **Recommendation:** Run a lightweight `SELECT 1` against `app.state.db_pool` (and/or the checkpointer pool) so readiness reflects real DB health.

### 🟢 Low

#### L1 — `CORS allow_credentials=True` is unnecessary
- **Location:** `apps/backend/src/api/main.py:104-110`
- **Detail:** The frontend authenticates with `Authorization: Bearer <jwt>`, not cookies (`apps/frontend/lib/api.ts`). Credentialed CORS isn't needed and slightly widens the surface.
- **Recommendation:** Set `allow_credentials=False` (the allowlist + bearer tokens remain fully functional).

#### L2 — Frontend route is client-guarded only
- **Location:** `apps/frontend/app/chat/page.tsx` → `ChatWorkspace`
- **Detail:** The chat shell renders before the session check; harmless because the backend enforces JWT on every request, but a Next middleware redirect for unauthenticated users would be cleaner UX and avoids a flash of protected UI.
- **Recommendation:** Optional — add `middleware.ts` to redirect unauthenticated users to `/login`.

#### L3 — Dead prompt versions retained
- **Location:** `apps/agents/src/agents/prompts/system.py` — `GENERATE_SYSTEM_V1`, `GENERATE_SYSTEM_V2`
- **Detail:** Only `GENERATE_SYSTEM_V3` is wired into `nodes.py`. V1/V2 are unused at runtime (V1 is still referenced by a test). They read as deliberate version history with rationale comments, so this is a style choice, not a bug.
- **Recommendation:** Either keep them intentionally, or move the evolution notes to a `prompts/CHANGELOG.md` and drop the dead constants (and update `test_prompts.py`).

#### L4 — Stale, misleading `prev-review/` documents
- **Location:** `prev-review/*.md`
- **Detail:** These review a different project (PO Parser / Image Tagger / Airtable / Google Apps Script). None of their file paths or findings map to this repo. Leaving them at the top level invites confusion.
- **Recommendation:** Delete them or move to an `archive/` folder clearly marked as unrelated.

## 5. What's genuinely good (keep doing this)

- **Clean layering.** `apps/agents` owns graph/nodes/tools/prompts/ingestion; `apps/backend` owns transport (routes/deps/auth/schemas/services); no cross-app leakage; the backend imports the compiled graph, not agent internals.
- **LangGraph best practices.** Deterministic node-based graph (not a black-box ReAct agent), narrow `TypedDict` state updates, per-node `RetryPolicy`, documented reducers (`add_messages`, `operator.add`), fail-closed generation (`EMPTY_CONTEXT_REPLY` when context is empty).
- **Clients are cached/pooled.** `@lru_cache` factories for the chat model, embedder, Neo4j driver, and Postgres pools — no per-request client construction. Every LLM call sets `timeout` and `max_tokens`.
- **Real GraphRAG.** Hybrid vector + full-text seed retrieval expanded via graph traversal (`MOVIE_CONTEXT_QUERY`), robust Text2Cypher with a bounded self-correction loop, a reranker, and a recommendation fallback so open-ended asks still ground in real movies.
- **Typed API contract.** `response_model` on the JSON routes, Pydantic `Field(description=...)` everywhere (feeds OpenAPI + env docs), request size caps (`max_length=2000` on chat message).
- **Async correctness.** Sync checkpointer/store work is dispatched via `run_in_threadpool` / `iterate_in_threadpool`, keeping the event loop responsive and working on Windows.
- **Ingestion quality.** Idempotent `MERGE` upserts, batched writes, manifest verification against loaded counts, Aura-Free limit guard, validated index identifiers, `tenacity` retries on embedding calls.
- **Docstrings + tests.** Google-style docstrings enforced in CI-ready lint; 71 tests covering pure functions, node wiring, route contracts, and error paths.
